🤖 ¿Qué es GitHub Copilot?

Asistente de IA de GitHub integrado directamente en VS Code. Entiende el contexto de tu proyecto, responde preguntas, propone cambios, ejecuta tareas y se conecta con herramientas externas — sin salir del editor.

Copilot Completions
Sugerencias de código en línea mientras escribes. Se activa solo.
Copilot Chat
Conversación en el panel lateral o inline (Ctrl+I). Tres modos: Ask, Plan, Agent.
Contexto personalizable
Custom instructions, prompt files y MCP amplían lo que Copilot sabe y puede hacer.

Fundamentos

Chat · Modos · Custom Instructions
💬
Chat
De prompt vago a prompt preciso
›

El chat de Copilot es la puerta de entrada. Pero un prompt vago produce una respuesta vaga. La diferencia está en el contexto que le das.

Abre el panel de Copilot Chat en VS Code haciendo clic en el ícono de Copilot en la barra de título, o usa el inline chat con Ctrl+I (Mac: ⌘I) directamente en el editor.

Prompt vago — resultado impredecible:

Copilot Chat
Crea Terraform para un archivo.

Prompt preciso con contexto — resultado consistente:

Copilot Chat
Crea un proyecto Terraform con provider hashicorp/local que:
- Genere un archivo JSON llamado config.json con campos: nombre_app, ambiente, version
- Use variables.tf con description en cada variable
- Incluya outputs.tf con la ruta del archivo generado
- Nombres en español, Terraform 1.5 o superior
Entrega main.tf, variables.tf y outputs.tf separados.
El prompt preciso produce archivos completos y bien estructurados desde la primera respuesta. Menos correcciones manuales.

Usa variables de contexto para resultados aún mejores:

Para adjuntar un archivo escribe # en el chat → selecciona el archivo del picker → Copilot inserta la referencia automáticamente. También puedes arrastrarlo al chat.

Copilot Chat — adjunta main.tf con el picker (#)
Revisa este archivo y asegúrate de que todas las variables tengan description y que los outputs tengan description también.
El chat mostrará la referencia como #file:main.tf automáticamente después de seleccionar con el picker. No se escribe el path a mano — se elige desde la UI.
Variables de contexto más usadas: #file (adjunta un archivo via picker), #selection (lo seleccionado en el editor), @workspace (búsqueda en todo el proyecto).
Guía del presentador
"El mismo Copilot, dos respuestas completamente distintas. La diferencia no es Copilot — es el prompt."

Ejecutar los dos prompts en vivo y comparar resultados. Mostrar cómo #file adjunta el contenido real del archivo al chat.

Pregunta: "¿Qué cambia cuando agregan contexto? ¿Qué más podrían agregar?"

Modos
Ask, Plan y Agent — cuándo usar cada uno
›

Copilot tiene tres modos de trabajo según la documentación oficial. Elegir el correcto marca la diferencia entre una respuesta útil y una pérdida de tiempo.

🔍
Ask
Solo lectura. Analiza código, responde preguntas, explica conceptos. No modifica archivos.
📋
Plan
Crea un plan detallado antes de actuar. Solo lectura hasta que apruebes el plan. Ideal para tareas complejas.
🤖
Agent
Ejecuta tareas autónomas: crea, edita y borra archivos, corre comandos. Ideal para flujos completos.

Cambia de modo con el selector en la parte inferior del chat o el Command Palette.

Ejemplo en Ask — para analizar sin riesgo:

Copilot Chat — modo Ask
@workspace ¿Qué variables de Terraform están definidas pero no se usan en ningún recurso? Lista los archivos y líneas.

Ejemplo en Agent — para flujo completo:

Copilot Chat — modo Agent
Crea la estructura completa de un proyecto Terraform con provider local:
- main.tf, variables.tf, outputs.tf
- .github/copilot-instructions.md con reglas básicas
- README.md con descripcion, variables, outputs y comandos de uso
Luego ejecuta terraform init y terraform validate para confirmar que está bien.
Agent mode crea todos los archivos y ejecuta los comandos Terraform sin intervención manual. Muestra el resultado en el chat.
Documentación oficial: docs.github.com/en/copilot — sección "Using Copilot Chat in VS Code" → "Chat modes".
Guía del presentador
"Ask es leer. Plan es planificar. Agent es actuar. Cada modo tiene su momento."

Mostrar el selector de modos en la UI. Correr el ejemplo de Agent en vivo para que vean los archivos crearse solos.

Pregunta: "¿Cuándo usarían Ask en vez de Agent? ¿Qué pasa si Agent se equivoca?"
~5 min

Custom Instructions
Reglas permanentes para todo el repositorio
›

Sin instrucciones, Copilot responde en inglés, usa nombres genéricos y sigue sus propias convenciones. Con copilot-instructions.md le das las reglas del equipo de forma permanente.

Crear el archivo: .github/copilot-instructions.md

.github/copilot-instructions.md
# Instrucciones del proyecto

- Responde siempre en español.
- Explica los cambios con lenguaje para principiantes.
- Usa nombres descriptivos en español para variables y recursos.
- Para Terraform: usa provider hashicorp/local (sin cloud).
- Todas las variables de Terraform deben tener description.
- Los outputs de Terraform deben tener description.
- Antes de proponer cambios grandes, explica qué vas a hacer y por qué.
resource "local_file" "f" { filename = "out.txt" content = var.c }
resource "local_file" "archivo_config" { filename = var.nombre_archivo content = var.contenido_archivo }

Prueba que las instrucciones están activas:

Copilot Chat
Agrega un nuevo recurso local_file a main.tf para crear un archivo de logs.
Copilot aplica las reglas sin que las repitas: nombres en español, variables con description, explicación en español. Automático.
Las instrucciones se aplican a todos los miembros del equipo que clonen el repo. Es la forma de estandarizar cómo trabaja Copilot para todos.
Guía del presentador
"No le repetí las reglas. Las leyó del repositorio. Cada respuesta las sigue automáticamente."

Mostrar la diferencia antes/después en la respuesta de Copilot. Destacar que esto se versiona en Git junto con el código.

Pregunta: "¿Qué reglas agregarían para su equipo? ¿Framework, idioma, convenciones de testing?"

Personalización

Instructions por archivo · Prompt Files · Contexto
🎯
Instructions por archivo
Reglas específicas según el tipo de archivo
›

Las instrucciones globales aplican a todo. Pero puedes crear instrucciones que solo se activan cuando Copilot trabaja con ciertos archivos, usando el campo applyTo en el frontmatter.

Crear carpeta y archivo: .github/instructions/terraform.instructions.md

.github/instructions/terraform.instructions.md
---
applyTo: "**/*.tf"
---

## Reglas para archivos Terraform

- Usa provider hashicorp/local (sin cloud).
- Estructura obligatoria: main.tf, variables.tf, outputs.tf.
- Variables: siempre con `type`, `description` y `default` si aplica.
- Outputs: siempre con `description`.
- Naming: snake_case en español para todos los identificadores.
- Bloque `terraform {}` con `required_providers` y `required_version`.
- Nunca valores hardcodeados: usar variables o locals.

Puedes tener una instruction por tipo de archivo o tecnología:

1 terraform.instructions.md → applyTo: "**/*.tf" ✓
2 markdown.instructions.md → applyTo: "**/*.md" ✓
3 python.instructions.md → applyTo: "**/*.py" ✓
4 github-actions.instructions.md → applyTo: ".github/workflows/*.yml" ✓

Prueba que la instrucción específica de Terraform se activa:

Copilot Chat — con un .tf abierto en el editor
Agrega una variable nueva llamada ambiente con valores permitidos dev, staging y prod.
Copilot genera la variable con type, description, default y una validation block con condition que restringe los valores — exactamente lo que dicta la instruction del archivo .tf.
Documentación oficial: Customizing Copilot → Custom instructions → Scoped instructions. El campo applyTo acepta glob patterns estándar.
Guía del presentador
"La instrucción global dice el idioma. La instrucción del archivo dice las reglas técnicas específicas."

Abrir un .tf y uno .md en paralelo. Mostrar que en el .tf aplican las reglas de Terraform y en el .md aplican otras reglas diferentes.

Pregunta: "¿Qué rules agregarían para los workflows de GitHub Actions? ¿Para los tests?"

Prompt Files
Recetas reutilizables para tareas comunes
›

Un prompt que tardaste 10 minutos en refinar no debería vivir solo en tu historial de chat. Los Prompt Files son prompts reutilizables versionados en el repo.

Crear: .github/prompts/crear-terraform.prompt.md

.github/prompts/crear-terraform.prompt.md
---
description: "Genera archivos Terraform completos desde una descripción"
agent: "agent"
---

El usuario te dará una descripción de lo que quiere crear con Terraform.

Antes de generar código:
1. Usa MCP de Terraform si está disponible para validar estructura.
2. Confirma que el provider sea hashicorp/local.

Genera los siguientes archivos:
- main.tf con provider y recursos
- variables.tf con todas las variables con description
- outputs.tf con todos los outputs con description

Explica cada bloque en español para principiantes.
Termina con los comandos de validación a ejecutar.

Crea un segundo prompt para documentación:

.github/prompts/generar-readme.prompt.md
---
description: "Genera README.md completo para un módulo Terraform"
agent: "ask"
---

Analiza los archivos .tf del proyecto actual y genera un README.md que incluya:

1. **Descripción** — qué hace este módulo en una línea.
2. **Prerrequisitos** — versiones de Terraform y providers.
3. **Variables** — tabla con nombre, tipo, descripción y default.
4. **Outputs** — tabla con nombre y descripción.
5. **Uso básico** — ejemplo de código HCL mínimo.
6. **Comandos** — init, validate, plan, apply, destroy.

Lenguaje: español. Tono: claro para principiantes.

Para usar un Prompt File, escribe su nombre en el chat precedido de / o selecciónalo desde el selector de prompts:

Copilot Chat
/crear-terraform Crea un módulo para generar un archivo JSON de configuración con campos: nombre_app, ambiente, version y puerto.
Copilot ejecuta el prompt con todo el contexto predefinido. Sin repetir instrucciones, sin olvidar reglas. Consistente entre todos los miembros del equipo.
El campo agent del frontmatter puede ser ask, agent o plan. Elige según el nivel de autonomía que quieres para esa tarea.
Guía del presentador
"El prompt que tardé 10 minutos en refinar ahora es un archivo en el repo. Cualquier dev del equipo lo usa con /crear-terraform."

Mostrar el selector de prompts en la UI. Ejecutar /crear-terraform en vivo y comparar con el mismo pedido sin el prompt file.

Pregunta: "¿Qué tareas repetitivas de su equipo se convertirían en prompt files?"

Variables de contexto
Dile exactamente a Copilot qué mirar
›

Copilot no lee todo el proyecto en cada pregunta. Tú le indicas qué contexto necesita con variables. Esto mejora la precisión y reduce respuestas irrelevantes.

# #file — escribe # → picker → seleccionas el archivo → inserta la referencia automáticamente ✓
# #selection — usa lo que tienes seleccionado en el editor ✓
# #sym — incluye el símbolo actual (función, clase, variable) en el contexto ✓
@ @workspace — busca en todo el proyecto (más lento, más completo) ✓
@ @github — busca en repositorios, issues y PRs de GitHub ✓

Comparar dos archivos — adjunta cada uno con el picker (#) y luego escribe la pregunta:

Copilot Chat — adjunta main.tf y variables.tf con el picker
¿Hay variables definidas en variables.tf que no se usan en main.tf? Lista cuáles son.

Revisar selección específica — selecciona el bloque en el editor, luego en el chat:

Copilot Chat — con código seleccionado en el editor
Explica qué hace este bloque de Terraform y si hay alguna mejora posible.

Buscar en todo el proyecto:

Copilot Chat
@workspace ¿En qué archivos .tf se usa el recurso local_file y cuántas instancias hay?
Respuestas precisas porque Copilot tiene exactamente el contexto relevante, no más ni menos. Menos alucinaciones, más utilidad.
#file siempre se selecciona desde el picker o arrastrando el archivo al chat. Lo que ves en el input (#file:main.tf) es la referencia que Copilot inserta automáticamente, no algo que se escribe a mano.
Usa #file (picker) para archivos concretos que conoces. Usa @workspace cuando no sabes en qué archivo está lo que buscas.
Guía del presentador
"No le dije 'mira el proyecto'. Le dije exactamente qué mirar. La respuesta es mejor porque el contexto es mejor."

Hacer en vivo la comparación: misma pregunta con @workspace vs con #file específico. Mostrar diferencia de velocidad y precisión.

Extensibilidad

MCP · Agent mode avanzado

MCP
Conecta Copilot con herramientas externas
›

Copilot por defecto no sabe qué versión tiene hoy el provider de Terraform. Con MCP le das acceso a fuentes externas en tiempo real: registries, APIs, bases de datos, documentación.

Configurar MCP en VS Code creando el archivo .vscode/mcp.json en el repo. VS Code detecta el archivo y muestra el botón Start junto al bloque de configuración:

.vscode/mcp.json
{
  "servers": {
    "terraform": {
      "command": "docker",
      "args": ["run", "-i", "--rm", "hashicorp/terraform-mcp-server"],
      "type": "stdio"
    }
  }
}

Pasos para agregar el servidor MCP desde la UI:

1 Crear .vscode/mcp.json con la configuración del servidor ✓
2 Guardar el archivo → VS Code muestra el botón Start junto al bloque de servidores ✓
3 Clic en Start → VS Code descubre las herramientas del servidor ✓
4 Abrir Copilot Chat → seleccionar modo Agent ✓
5 Clic en el ícono de herramientas para ver los servidores activos ✓

Ahora usar MCP en Agent mode para pre-revisión:

Copilot Chat — modo Agent + MCP activo
Usa MCP de Terraform para revisar el proyecto actual.
Detecta:
- Variables definidas pero no usadas en ningún recurso
- Referencias a recursos que no existen en el proyecto
- Outputs que referencian atributos incorrectos
- Buenas prácticas que faltan (description, required_version)
Dame un reporte con severidad: ERROR, WARNING, INFO.
MCP consulta la estructura del Terraform actual y el Registry en tiempo real. Detecta problemas antes de que terraform validate los encuentre — con mayor contexto y explicaciones.

Usar MCP para generar con documentación actualizada:

Copilot Chat — modo Agent
Consulta el MCP de Terraform para obtener la documentación actualizada del recurso local_file del provider hashicorp/local. Luego genera un ejemplo completo con todos los argumentos opcionales documentados.
Requisito: Docker instalado y corriendo. El servidor MCP de Terraform es oficial de HashiCorp. Hay servidores MCP para GitHub, Atlassian, bases de datos, AWS Docs y más en el marketplace de VS Code.
MCP no reemplaza terraform validate. MCP complementa el análisis antes de llegar a Terraform CLI. El flujo correcto: MCP → validate → plan → apply.
Guía del presentador
"Documentación oficial del provider, leída en tiempo real desde el Registry. No de su entrenamiento de hace meses."

Mostrar el panel MCP en VS Code y el servidor activo. Si alguien no tiene Docker, mostrar el resultado esperado sin ejecutar.

Pregunta: "¿Qué otros servidores MCP les serían útiles? ¿GitHub para ver issues? ¿Confluence para leer specs?"

Laboratorio Final

Instructions · Prompts · MCP · S3 · MiniStack — flujo completo
🌐
Lab Final
Todo junto: Instructions + Prompts + MCP + S3 con MiniStack
›

El laboratorio final integra todo lo aprendido en el workshop: instrucciones globales (copilot-instructions.md), instrucciones por tipo de archivo con applyTo, prompt files reutilizables, MCP para documentación en tiempo real y Agent mode. MiniStack emula S3 localmente — gratuito, sin cuenta, sin API key.

¿Qué es MiniStack? Alternativa open-source (MIT) a LocalStack. 60+ servicios de AWS en un solo contenedor Docker. Compatible con el provider hashicorp/aws de Terraform sin modificaciones. Arranque en ~2 segundos, ~30 MB RAM. ministack.org

Paso 0 — Levantar MiniStack:

Terminal
docker run -d -p 4566:4566 --name ministack ministackorg/ministack
Mismo puerto 4566 que LocalStack — drop-in replacement. Sin cuenta, sin API key, sin telemetría. Verificar: curl http://localhost:4566/_ministack/health

Paso 1 — Crear instrucciones globales del proyecto:

Antes de generar código, configuramos el contexto de Copilot. Estas reglas aplican a todas las conversaciones del repo desde que se guarda el archivo:

.github/copilot-instructions.md
# Reglas del Lab — Terraform + MiniStack

- Responde siempre en español
- Usamos MiniStack como emulador local de AWS (puerto 4566)
- El provider AWS apunta a http://localhost:4566 — nunca a AWS real
- Credenciales de prueba: access_key = "test", secret_key = "test"
- Los buckets S3 son para web estática pública (block_public_acls = false)
- Explica cada bloque de Terraform para principiantes
- Nombres de recursos en snake_case
Guardado en .github/copilot-instructions.md. Copilot aplica estas reglas en cada conversación del repo — español, MiniStack, credenciales de prueba y explicaciones para principiantes.

Paso 2 — Instrucciones específicas para archivos Terraform:

Con applyTo, estas reglas solo se activan cuando Copilot trabaja sobre archivos .tf. Precisión sin ruido en otros tipos de archivo:

.github/instructions/terraform.instructions.md
---
applyTo: "**/*.tf"
---

# Convenciones Terraform — Lab MiniStack

- Todas las variables y outputs tienen `description`
- El provider aws siempre incluye `skip_credentials_validation = true`
- Los buckets de laboratorio tienen `force_destroy = true`
- `depends_on` explícito cuando aws_s3_bucket_policy depende de aws_s3_bucket_public_access_block
- Terraform >= 1.5, provider hashicorp/aws ~> 5.0
- Sin módulos externos — archivos planos del laboratorio
Guardado en .github/instructions/terraform.instructions.md. Cada vez que Copilot trabaje sobre un .tf, aplicará force_destroy, description en variables y depends_on correcto — sin pedírselo.

Paso 3 — Crear el prompt file del despliegue:

Encapsulamos toda la lógica en un prompt file versionado en el repo. Cualquier miembro del equipo lo invoca con /desplegar-web-s3 — sin repetir instrucciones:

.github/prompts/desplegar-web-s3.prompt.md
---
description: "Despliega una web estática en S3 con MiniStack"
agent: "agent"
---

Usa MCP de Terraform para consultar la documentación actualizada del provider hashicorp/aws.

Genera estos archivos en la carpeta lab-s3-ministack/:

**providers.tf** — provider aws apuntando a MiniStack:
- endpoint S3: http://localhost:4566
- region us-east-1, access_key y secret_key = "test"
- skip_credentials_validation = true, skip_requesting_account_id = true

**main.tf** — recursos en orden:
1. aws_s3_bucket (force_destroy = true)
2. aws_s3_bucket_website_configuration (index_document: index.html)
3. aws_s3_bucket_public_access_block (todos los flags en false)
4. aws_s3_bucket_policy (GetObject público, depends_on en el bloque anterior)
5. aws_s3_object (sube web/index.html, content_type = "text/html")

**variables.tf** — variable nombre_bucket con description y default "hola-terraform-ia"
**outputs.tf** — output url_web con la URL del objeto en MiniStack

Explica cada recurso en español para principiantes al terminar.
Prompt file listo en .github/prompts/desplegar-web-s3.prompt.md. Disponible como /desplegar-web-s3 en cualquier chat de Copilot dentro del repo. Copilot ya tiene todo el contexto — instrucciones globales, reglas por archivo y la lógica del despliegue.

Paso 4 — Invocar el prompt file con MCP activo:

Con las instrucciones y el prompt file configurados, una sola línea es suficiente. Copilot ya conoce el contexto completo:

Copilot Chat — modo Agent + MCP Terraform activo
/desplegar-web-s3
El prompt file ya tiene todo el contexto: provider, endpoint MiniStack, estructura de archivos, orden de recursos y convenciones en español. El MCP trae la documentación más reciente del provider. Una línea reemplaza un prompt de 15 líneas — y es consistente para todo el equipo.

Copilot genera los cuatro archivos con la documentación más reciente del provider:

lab-s3-ministack/providers.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_requesting_account_id  = true
  skip_metadata_api_check     = true
  endpoints {
    s3 = "http://localhost:4566"
  }
}
lab-s3-ministack/main.tf
resource "aws_s3_bucket" "web" {
  bucket        = var.nombre_bucket
  force_destroy = true
}

resource "aws_s3_bucket_website_configuration" "web" {
  bucket = aws_s3_bucket.web.id
  index_document {
    suffix = "index.html"
  }
}

resource "aws_s3_bucket_public_access_block" "web" {
  bucket                  = aws_s3_bucket.web.id
  block_public_acls       = false
  block_public_policy     = false
  ignore_public_acls      = false
  restrict_public_buckets = false
}

resource "aws_s3_bucket_policy" "web" {
  bucket     = aws_s3_bucket.web.id
  depends_on = [aws_s3_bucket_public_access_block.web]

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = "*"
      Action    = "s3:GetObject"
      Resource  = "${aws_s3_bucket.web.arn}/*"
    }]
  })
}

resource "aws_s3_object" "pagina" {
  bucket       = aws_s3_bucket.web.id
  key          = "index.html"
  content      = file("${path.module}/web/index.html")
  content_type = "text/html"
}
lab-s3-ministack/variables.tf
variable "nombre_bucket" {
  description = "Nombre del bucket S3 para la web estática"
  type        = string
  default     = "hola-terraform-ia"
}
lab-s3-ministack/outputs.tf
output "url_web" {
  description = "URL de la web estática en MiniStack"
  value       = "http://localhost:4566/${var.nombre_bucket}/index.html"
}
4 archivos Terraform generados con las convenciones del lab: variables con description, force_destroy = true y depends_on correcto en la policy. Todo gracias a las instrucciones de applyTo — sin pedirlo explícitamente.

Paso 5 — Crear la página web con Agent mode:

Copilot Chat — modo Agent
Crea el archivo lab-s3-ministack/web/index.html. El contenido debe decir:
"Hola, soy Terraform potenciado con IA 🤖"

Paso 6 — Desplegar con Terraform:

Copilot Chat — modo Agent
Entra a la carpeta lab-s3-ministack, ejecuta terraform init y terraform apply -auto-approve. Luego muestra el valor del output url_web.
1 terraform init — descarga el provider hashicorp/aws ✓
2 terraform apply -auto-approve — crea bucket, policy y sube el HTML ✓
3 Abrir http://localhost:4566/hola-terraform-ia/index.html en el navegador ✓
4 Limpiar: terraform destroy -auto-approve → docker stop ministack ✓

La fórmula completa

GitHub Copilot según la documentación oficial
Ask mode        = analizar y responder     → sin modificar archivos
Plan mode       = planear antes de actuar  → apruebes, luego edita
Agent mode      = ejecutar tareas          → crea, edita, corre comandos

copilot-instructions.md  = reglas para todo el repo  → siempre activas
instructions/*.md        = reglas por tipo de archivo → activadas por applyTo
prompts/*.md             = recetas reutilizables     → invocadas con /nombre

#file     = picker → adjunta archivo   → contexto específico
#selection= lo seleccionado           → contexto mínimo
@workspace= todo el proyecto          → contexto máximo

MCP       = herramientas externas     → docs, registries, APIs en tiempo real
